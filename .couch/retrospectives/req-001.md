# Retrospective: req-001 + req-001-fix-1 + req-002

## Summary

Three requirements ran in this session, all ultimately accepted. req-001 (5 tasks, 3 waves) rewrote the Team Lead skill in AGENTS.md operational-manual style and added harness enforcement via PreToolUse hook scripts. req-001-fix-1 was a Correction Mode run that patched a critical hook path bug missed during the original execution. req-002 codified three spawn/reuse protocol gaps that surfaced from the req-001 retrospective discussion. The work was substantively correct and the deliverables are high quality. The session revealed two systemic gaps — one in how Coder acceptance criteria specify verification scope for harness tasks, and one in how disposition tables handle "move this constraint to file X" decisions — and two environmental incidents (Team Lead model non-determinism and harness team state loss). The acceptance judge layer proved its value by catching the hook path bug before shipping; without it, broken enforcement would have been silently deployed.

## Friction Analysis

### Friction 1 — Coder shipped relative hook paths that silently fail at runtime

**What happened**: task-002 spec contained an explicit escalation trigger: "If you discover... PreToolUse hooks in SKILL.md frontmatter cannot reference a relative script path (must be absolute or inline) — STOP and escalate to Architect before continuing." The Coder wrote bare relative paths (`hooks/restrict_write_path.sh`) and ran all 17 unit tests by piping JSON directly into the script executables. Every test passed because the scripts themselves are correct. The hook invocation mechanism — whether Claude Code would actually call the script when the frontmatter fires — was never exercised. The bug was caught only by acceptance-review judges cross-referencing the official Claude Code docs and GitHub issues.

**Classification**: Systemic

**Root cause**: The Coder SOUL ("Pragmatic Craftsperson") and coder.md Action Framework both emphasize "small verifiable increments" and "evidence of verification," but neither specifies that when the acceptance criterion is about harness or system enforcement, verification must exercise the enforcement mechanism end-to-end, not just the component in isolation. The Action Framework's step 5 says "implement in smallest verifiable increments" without distinguishing between component-level and integration-level verification targets. When the target is "this hook fires correctly when Claude Code invokes it," running the script directly is not a valid substitute — but nothing in the coder definition signals this distinction.

**Root cause file**: `.claude/agents/coder.md` — Action Framework step 5 and the Self-Verification / Evidence standards section.

**Note (first occurrence)**: No prior retrospective exists to confirm a second occurrence. Noting for future reference. If this pattern appears again (Coder runs component tests as evidence for a criterion that requires integration/harness verification), propose adding an explicit note to coder.md.

### Friction 2 — Disposition table accepted a "DELETED / belongs in protocol.md" decision without verifying the destination already contained the rule

**What happened**: In task-001, constraint #16 ("NEVER downgrade an agent's model below its frontmatter default. For Coder tasks with complexity L, upgrade to opus at spawn time") was marked DELETED with the justification "narrow spawn-time config detail; belongs in protocol.md Spawn Prompt Template." Three reviewers (Architect + two judges) accepted this disposition. Nobody grepped protocol.md to verify the rule existed there. It did not. The rule was silently lost. req-002 was required to recover it.

**Classification**: Systemic

**Root cause**: The disposition methodology in task-001's acceptance criteria says to provide a table "with a one-line justification for EVERY entry" and "label deleted / tooling / workflow / situation." It does not require verification that destination files for "belongs elsewhere" justifications actually contain the referenced content. The Architect task spec and the coder.md definition both lack any "verify destination before deleting" discipline. This is a structural gap: the system trusts the word "belongs" without a confirmation step. The same risk applies to any future constraint or rule migration between SKILL.md, protocol.md, workflow.md, or any reference doc.

**Root cause file**: `.couch/requirements/req-001/tasks.json` task-001 acceptance criteria (source of the disposition methodology as specified in this run); more broadly, this methodology would need to be encoded in `.claude/agents/architect.md` or the relevant SOUL if it recurs.

**Note (first occurrence)**: This is the first occurrence. The gap is structurally significant — any disposition that says "deleted, belongs in X" is a latent hole unless the destination is verified. However, per retrospective protocol, a proposal requires 2+ occurrences across runs. Noting for future reference.

### Friction 3 — Team Lead did not pass `model:` explicitly when spawning agents

**What happened**: Team Lead spawned all agents across req-001, req-001-fix-1, and req-002 without passing an explicit `model:` parameter to the Agent tool. This means each spawn was subject to the Opus orchestrator inheritance hazard: when an Opus Team Lead calls Agent without `model:`, the harness non-deterministically (~33% probability) inherits the orchestrator's model rather than reading the agent frontmatter. Some fraction of "sonnet" coders in this session likely ran on Opus. Team Lead did not know about this hazard until Researcher documented it in req-002.

**Classification**: Exploratory (discovery was not possible from information available before this run)

**Root cause**: The model tier resolution behavior and Opus inheritance hazard were not documented in protocol.md at the start of this session. They were a true unknown-unknown — Researcher sourced them from official Claude Code docs and a GitHub issue (#32732) that was not previously reflected in any system documentation. There was no file or SOUL that Team Lead could have consulted to learn this.

**Why exploratory**: The system had no information that could have prevented this. The knowledge existed only externally (Claude Code docs + GitHub). req-002 has now closed this gap by adding the hazard documentation and mandatory `model:` rule to protocol.md. Future Team Leads reading protocol.md will see it before spawning.

**Note**: req-002 has already applied the fix. No further proposal needed. Noting that the fix is in `templates/skill/references/protocol.md` Spawn Prompt Template section.

### Friction 4 — Team state loss during the session (TeamCreate/spawn loop)

**What happened**: Twice during the session, spawning a teammate via the Agent tool with `team_name: req-001` returned "Team 'req-001' does not exist" despite Team Lead already being team-lead of that team. Workaround required TeamDelete + TeamCreate, which cleared TaskList and required task reconstruction.

**Classification**: Exploratory (external harness behavior; not actionable from this project's definition files)

**Root cause**: This is a Claude Code harness bug, not a gap in Couch Potato's agent definitions, SOULs, or reference documentation. The symptom is team state being unexpectedly invalidated during a long session. Couch Potato cannot prevent this via its own configuration.

**Note (first occurrence)**: First occurrence in this project. No proposal possible — the bug is not in this system's files. If it recurs, the pattern worth documenting is: after team state loss, Team Lead should verify team state (read `~/.claude/teams/{req-id}/config.json`) before reconstructing tasks, to confirm what is actually lost. This could be added as a note to protocol.md Initialization section if the pattern occurs again.

## Tool Usage

**Coder (task-001)**: Performed the mandatory pre-flight grep before writing any files. Documented all 19 constraints verbatim with justifications. Tool usage was disciplined and evidence-based.

**Coder (task-002)**: Used shell execution to run 17 unit tests and documented all results in a table. Evidence was real (observed exit codes), not self-assessment. However, the tool used (direct script invocation) was not the right tool for the acceptance criterion being verified (harness invocation). This is the core of Friction 1 — not a failure to use tools, but a failure to use the correct test methodology for the criterion type.

**Coder (fix-001)**: Added a Phase 2 test suite that invoked the exact SKILL.md command string via `sh -c` with `CLAUDE_PROJECT_DIR` set — exactly the methodology that task-002's Coder should have used. The fix Coder identified and filled the gap in test methodology.

**Judges (req-001 review)**: Cross-referenced official Claude Code docs and GitHub issues rather than relying on training data. This is the expected discipline from the SOUL design, and it worked — external evidence caught the relative path bug that unit tests could not.

**Researcher (req-002)**: Sourced model tier resolution facts from official docs and a specific GitHub issue (#32732). No training-data reliance detected.

## SOUL Health

Four SOULs were active in this session: Coder (Pragmatic Craftsperson), Tester (Suspicious Verifier), Researcher (no named SOUL visible in this session), and the judges (operating with Architect + Researcher SOULs).

**Convergence check**: The Coder SOUL ("simplest thing that works correctly") and Tester SOUL ("prove it works; I don't believe you until I see evidence") are designed to be in tension. In task-002, that tension was not realized: the Tester accepted 17 unit tests as sufficient evidence for a criterion about harness invocation. The Tester SOUL says "external evidence over self-assessment" and "run it, screenshot it, measure it" — but the Tester did not distinguish between "running the script" and "running the harness that invokes the script." Both SOULs converged on "the script works" without either attending to "does the harness fire it."

This is the only convergence signal observed. The other agents (Architect, Researcher, judges) maintained distinct attention patterns: Architect attended to structural correctness, judges attended to integration-level verification against external sources, Researcher attended to factual accuracy from primary sources.

**Note**: The Tester SOUL's "What I attend to" list — "external evidence over self-assessment," "happy path baseline THEN edge cases" — does not currently include any discipline about matching test methodology to the level of integration the criterion requires. If Friction 1 recurs, this is a candidate addition to the Tester SOUL.

## Patterns Observed

1. **Harness-level acceptance criteria without harness-level test methodology** (Friction 1): When a task spec writes an acceptance criterion about system enforcement behavior (hooks firing, disallowedTools triggering), both Coder and Tester default to component-level verification. The component passes; the integration fails. First occurrence.

2. **"Belongs in X" dispositions without destination verification** (Friction 2): Disposition tables that mark constraints as DELETED with a "belongs elsewhere" justification create silent documentation holes unless the destination file is grepped before deletion. First occurrence.

3. **Harness state loss during long sessions** (Friction 4): Team state may be unexpectedly invalidated during extended runs in this harness version. First occurrence.

The two systemic frictions (Friction 1, Friction 2) are the highest-value candidates for proposals in the next retrospective if either pattern recurs:

- **Friction 1 candidate target**: `.claude/agents/coder.md` Action Framework step 5 — add a note that for tasks where the acceptance criterion involves harness/system enforcement, verification must exercise the enforcement mechanism end-to-end (not the isolated component). Candidate addition to `.claude/skills/couch-potato/references/souls/tester.md` "What I attend to" — match test methodology to the integration level the criterion requires.

## Clean Run Notes

- The acceptance judge layer is load-bearing for harness tasks. It caught a critical bug (silent hook path failure) that component testing and self-verification could not. The architecture of spawning independent judges with access to external docs is validated.
- req-002's one-task structure (single file, single section, all four additions) was efficient. Well-scoped requirements with tight file ownership produce clean execution.
- The Coder SOUL's pre-flight discipline (grep before write) in task-001 worked exactly as intended and surfaced a correct "all clear" before modifying files.
- Disposition tables with verbatim constraint quotation and per-entry justifications are a sound practice — they made req-001-fix-1 and req-002 triageable without re-reading the original SKILL.md.
