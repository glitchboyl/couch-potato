## Friction Analysis

### Friction 1 — Proposal D included in original scope despite being third coverage of the same invariant
- **What happened**: Architect's initial plan included task-001 (Proposal D from req-010 retro) as a Wave Exit Checklist pre-check. architect-challenger's stress-test identified it as redundant: the same invariant is already covered by Proposal A (SKILL-body.md "Before you do anything else") and protocol.md §"If team state appears inconsistent or spawn errors." Architect conceded after attempting to construct a unique failure mode and finding the argument did not hold.
- **Classification**: planning error (challenger-caught)
- **Root cause**: The req-010 retro characterized Proposal D as "defense-in-depth," which is a framing that makes redundancy sound virtuous. Architect accepted that framing without independently testing whether D catches anything A + protocol.md do not. The challenger's question — "name the concrete failure mode D uniquely catches" — is the right diagnostic and should be applied to any defense-in-depth proposal before writing the task.
- **Impact**: Low — caught in planning, before any Coder was dispatched. The cut saved one task with no production value.

### Friction 2 — task-002 acceptance criteria over-prescribed retro format
- **What happened**: The original acceptance criterion for task-002 specified that the bootstrap must be logged in retro as "authorized bootstrap override" with the authorization quote, what was done, and what verification passed — prescribing internal retro structure from a Situations bullet in SKILL-body.md. architect-challenger flagged this as overreach: retro format is Retrospective Agent's domain.
- **Classification**: planning error (challenger-caught)
- **Root cause**: Architect conflated "Team Lead should ensure the retro captures this clearly" with "Team Lead should specify what fields the retro uses." The correct boundary: Team Lead's instruction is what information must be present (authorization, action, verification outcome); Retrospective Agent's judgment is how to structure it.
- **Impact**: Low — acceptance criteria narrowed before Coder dispatch.

## Standing Diagnostic: Convergence Test for Retro Proposals

*Produced by architect + architect-challenger joint amendment, req-012 debate. Applies to all future retrospective proposals.*

**The test:**

> If a retro proposal is "add a rule that checks whether an earlier rule was followed, in a file the agent is already guaranteed to read," the proposal is marginal — skip it.
>
> If the rule exists only in a referenced-but-not-mandatory file, moving it into the mandatory read path is enforcement, not redundancy, and is worth landing.

**How to apply:**

Before writing a retro proposal, ask: "What file is the mandatory read path for the agent who needs this rule?"

- If the rule already exists in that file or in a file the agent is mandated to read before acting: the proposal is noise. Do not write it.
- If the rule exists only in a referenced-but-optional file: moving or adding it to the mandatory path is legitimate enforcement. Write it.

**Diagnostic question to run on any defense-in-depth proposal:**

> "Name the concrete failure mode this catches that the existing rules in the mandatory read path do not already address."

If you cannot name one, cut the proposal.

**Provenance:**

- architect-challenger's original test: "If the retro's proposal is 'add a rule that checks whether an earlier rule was followed,' that is the marginal finding."
- Architect's amendment: added the qualifying clause ("in a file the agent is already guaranteed to read") to preserve legitimacy of moving rules from referenced-but-optional files into the mandatory path (the Proposal A pattern from req-010).
- The amended form passed its own test: Proposal A from req-010 would have passed (moved rule from protocol.md — referenced-but-not-mandatory — into SKILL-body.md — mandatory entrypoint). Proposal D from req-010 failed it (triple coverage of the same invariant in files all in the mandatory path).
