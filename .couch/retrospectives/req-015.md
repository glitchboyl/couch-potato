# Retrospective: req-015

## Summary

req-015 closed two silent structural violations surfaced by a live first-use field report: (1) the init skill's Case B writing the agent-teams flag to the committed `settings.json` instead of the gitignored `settings.local.json`, and (2) the team-mode SKILL-body.md mandatory gate lacking explicit constraints on `team_name` value and `run.json` path, allowing Team Lead to silently diverge from the intended structure. Both fixes landed correctly. The planning phase also surfaced a significant process finding: three LLMs in succession accepted a factual claim about harness behavior without verifying it against current code — a sycophancy pattern that the adversarial review process did not catch. That pattern is this retro's most important output.

## Friction Analysis

### Friction 1 — Architect's initial four-identifier invariant was factually wrong about team_slug

- **What happened**: Architect's first version of task-002 included a "four-identifier invariant" asserting that `team_name`, `requirement_id`, `team_slug`, and the directory name under `.couch/requirements/` are "four spellings of the same string." Challenger correctly identified this as false: `team_slug` is the harness-assigned output that may differ from `req-id`, and was added to `run.json` precisely to record that divergence. Architect conceded and updated the wording to a three-identifier invariant with an explicit `team_slug` distinction.
- **Classification**: planning error (challenger-caught)
- **Root cause**: Architect did not read `run.json`'s existing `team_slug` field with enough attention to its origin. The field was added by req-009 Proposal A specifically because the harness ignores the requested `team_name` — but Architect read the field as a synonym for the requirement ID rather than a potentially-divergent harness output.
- **Impact**: Low — caught in planning before any Coder was dispatched. The final wording is correct.

### Friction 2 — Sycophancy in adversarial review: the team_slug claim and what should have happened

- **What happened**: Challenger's Defect 2 rested on the claim "the harness may ignore `team_name` and assign a random slug, per req-009 Friction 1 and protocol.md step 8." Architect accepted this without verification. Team Lead routed the concession without verification. Three LLMs in sequence took each other's word on a factual claim about current harness behavior and current protocol.md text.
- **What verification would have revealed**:
  - **req-009 Friction 1 does support the claim** — the retro documents exactly this behavior with the example slug `delegated-frolicking-stonebraker`, and Proposal A documented the fix to add advisory-language to protocol.md.
  - **BUT Proposal A from req-009 was never landed** — the current `references/team-mode/protocol.md` step 8 reads word-for-word its pre-req-009 form: "TeamCreate with `team_name: <req-id>` — this creates the shared task list and messaging channel for the swarm." No advisory language, no random-slug mention, no `team_slug` capture instruction. The proposal was accepted but not applied.
  - **AND session evidence contradicts the retro's description of current behavior** — Team Lead ran `TeamCreate` four times across req-011, req-012, req-013, and req-015. Every time the harness returned the exact `team_name` passed. Zero random slugs. `leadAgentId` was always `team-lead@req-XXX`, `config.json` was always at `~/.claude/teams/req-XXX/`. The current harness either does not do random-slug assignment, or does it non-deterministically in a way that was not triggered across four consecutive runs.
- **The chain**: Challenger cited a retro → Architect accepted → Team Lead routed → nobody verified. The adversarial review process collapsed into polite acknowledgment.
- **Product outcome**: The landed wording ("team_slug may differ") is a defensive hedge that is harmless under current harness behavior and correct if the historical behavior re-emerges. The product is fine. The process was broken.
- **Classification**: systemic (adversarial review process gap)
- **Root cause**: The adversarial review protocol has no verification requirement. When one architect cites a claim about codebase state, harness behavior, or protocol text, the receiving architect is expected to debate it — but there is no rule requiring direct verification (read the file, grep, recall session evidence) before accepting or rejecting. Acceptance by acknowledgment alone is sycophancy, and is structurally indistinguishable from the adversarial review working correctly.
- **Root cause file**: No current file documents this rule — the gap is its absence. The landing target for the fix is a future requirement (req-016).

### Friction 3 — Convergence test is a redundancy filter, not a correctness filter

- **What happened**: The req-012 convergence test passed both proposals ("rules exist only in referenced-but-not-mandatory file — moving them into mandatory path is enforcement, not redundancy"). The test did not catch Architect's factually wrong invariant in task-002 because the test only asks "is this rule already in the mandatory path?" — it does not ask "is this rule correct?"
- **Relationship to Friction 2**: the two are adjacent. A factually wrong rule can pass the convergence test and land in the mandatory path if the adversarial reviewer doesn't verify the underlying claim. The test is necessary but not sufficient for correctness.
- **Classification**: calibration note (not a standalone proposal — the fix is in the verification rule for adversarial review, which is Friction 2's proposal)
- **Impact**: Low in this instance — the landed wording is correct. The observation is about the test's scope, not its validity.

## Improvement Proposals

### Proposal A — Add fact-verification rule to adversarial review protocol

- **Target file**: To be determined in req-016 (likely `references/team-mode/SKILL-body.md` spawn instructions for temporary discussion agents, or a new adversarial-review protocol section)
- **Section**: Wherever architect-challenger spawn instructions live, or a new "Adversarial review standards" section
- **Proposed rule**:
  > When one architect cites a claim about codebase state, harness behavior, or protocol text, the receiving architect must verify the claim directly — read the current file, grep for the current text, or recall concrete session evidence — before accepting or rejecting. Acceptance by acknowledgment alone is sycophancy and is not valid adversarial review. "The retro says X" is not the same as "X is true now."
- **Rationale**: The req-015 debate produced a three-LLM acknowledgment chain on a claim that was (a) grounded in a retro but (b) not reflected in current protocol.md text and (c) contradicted by four consecutive sessions of session evidence. The product outcome was harmless; the process was indistinguishable from non-adversarial review. The rule closes the gap by requiring direct verification, not just openness to debate.
- **Note**: Do NOT attempt to land this rule as part of req-015. It requires its own requirement (req-016) with proper Architect planning and Challenger review — including, ideally, a Challenger who applies the new rule to the proposal itself.
